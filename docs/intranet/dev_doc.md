# Intranet Backend — Developer Documentation

## Overview

The Intranet backend is a **Python Flask API** located at `apps/intranet/backend/`. It follows the **application factory pattern**, connects to a PostgreSQL database via SQLAlchemy, and authenticates requests using Auth0 JWTs. It is designed to run inside Docker (via `apps/intranet/docker-compose.yml`) and can also be run locally for development.

Shared Python code lives in `libs/` as installable packages, keeping the backend thin and focused on application logic.


---

## API Versioning

All backend API endpoints are versioned and preceded by `/api/{version}`. 

Currently, the active version is **v2**. Therefore, every API endpoint is prefixed with `/api/v2/` (e.g., `/api/v2/auth/login`, `/api/v2/attendance/clock-in`). 

This prefixing is implemented dynamically via Flask's blueprint nesting in the application factory (`__init__.py`). All feature blueprints are registered as children under the parent `api_v2` blueprint, which is then registered directly to the application.

## Type Safety & Static Analysis

The backend codebase is fully type-annotated and uses **Mypy** for static type checking. 

### Mypy Configuration (`mypy.ini`)
The configuration enforces strict type checking rules:
* `check_untyped_defs = True` - Ensures type coverage on all functions.
* `warn_unused_ignores = True` and `warn_redundant_casts = True` - Keeps annotations clean.
* Excludes autogenerated files like migrations.
* Disables `name-defined` error codes for `models` and `backend.models` to support Flask-SQLAlchemy's dynamic runtime proxies while keeping the rest of the application strictly typed.

### SQLAlchemy 2.0 Mapped Types
Database models in `models.py` use SQLAlchemy 2.0's type-safe declarative mapping system:
* Columns are defined using `Mapped[Type]` and `mapped_column()`.
* Relationships are explicitly typed (e.g., `modules: Mapped[list[Module]]`).
* This enables mypy to verify type correctness across all route handlers, services, and queries.

To run the type check locally:
```powershell
.\backend\.venv\Scripts\python.exe -m mypy --config-file=backend/mypy.ini backend
```

## Application Factory

The app is initialised using the factory pattern in `__init__.py`. Nothing is created at import time — the app object is only created when `create_app()` is called.

```python
from __init__ import create_app
app = create_app()
```

**Why this matters:** The factory pattern makes the app testable (you can create multiple app instances with different configs), avoids circular imports, and is the standard Flask approach for non-trivial applications.

### `__init__.py` responsibilities

1. Inserts the backend directory into `sys.path` so sibling modules (`models`, `auth_routes`) can be imported with absolute names regardless of how Python loads the package.
2. Loads `py_config.get_config()` to configure the Flask app (database URI, debug mode).
3. Initialises extensions: `SQLAlchemy`, `Flask-Migrate`, `Flask-CORS`.
4. Registers the `auth` blueprint.
5. Registers a global `AppError` error handler that renders a consistent JSON error response for every route in the app.

## Shared Python Libraries

Three shared packages live under `libs/`. They are installed as **editable local packages** (`pip install -e`) so changes to their source are reflected immediately without reinstalling.

All three follow the same structure:

- Source files live directly at the root of the library folder (no nested package subdirectory).
- `pyproject.toml` uses `[tool.setuptools.package-dir]` to map the root `.` to the import name.

### `libs/py_errors` → `import py_errors`

Provides all shared exception classes used across backends.

**Base class:** `AppError(Exception)` — holds an error code, a human-readable message, an HTTP status code, and optional debug details. Renders a structured JSON body via `write_response()`:

```json
{
  "error": "<error_code>",
  "message": "<human-readable description>",
  "details": "<optional: shown in debug mode or for validation errors>"
}
```

**Available exception classes:**

| Class                         | HTTP | Error Code                 | Default Message                                 |
| ----------------------------- | ---- | -------------------------- | ----------------------------------------------- |
| `UnauthorizedError`           | 401  | `token_invalid`            | Missing or invalid token.                       |
| `ForbiddenError`              | 403  | `insufficient_permissions` | You do not have permission.                     |
| `NotFoundError`               | 404  | `not_found`                | Resource not found.                             |
| `EmployeeNotFoundError`       | 404  | `employee_not_found`       | Employee not found.                             |
| `DepartmentNotFoundError`     | 404  | `department_not_found`     | Department not found.                           |
| `DepartmentHasEmployeesError` | 409  | `department_has_employees` | Cannot delete department with active employees. |
| `AlreadyClockedInError`       | 409  | `already_clocked_in`       | You are already clocked in.                     |
| `NotClockedInError`           | 409  | `not_clocked_in`           | You are not clocked in.                         |
| `EmailExistsError`            | 409  | `email_already_exists`     | An employee with this email already exists.     |
| `ValidationError`             | 422  | `validation_error`         | Request body failed validation.                 |
| `InternalError`               | 500  | `internal_error`           | An unexpected error occurred.                   |

**Usage in a route:**

```python
from py_errors import EmployeeNotFoundError, ValidationError

employee = Employee.query.get(user_id)
if not employee:
    raise EmployeeNotFoundError()

if not role_id:
    raise ValidationError(details="role_id is required.")
```

The global error handler in `__init__.py` catches all `AppError` subclasses and calls `e.write_response()` automatically — routes never need to construct the JSON error response themselves.

---

### `libs/py_config` → `import py_config`

Provides Flask configuration classes based on the environment.

**`get_config()`** — returns the appropriate config class based on `FLASK_DEBUG`:

- `FLASK_DEBUG=1` → `DevConfig` (debug on)
- `FLASK_DEBUG=0` (or unset) → `ProdConfig` (debug off)

Both config classes read `DATABASE_URL` from the environment and set it as `SQLALCHEMY_DATABASE_URI`.

**Usage:**

```python
from py_config import get_config
app.config.from_object(get_config())
```

The `DATABASE_URL` environment variable must be set. See [Environment Variables](#environment-variables) below.

---

### `libs/py_auth` → `import py_auth`

Provides Auth0 JWT verification shared across all app backends.

**`verify_jwt(token, domain, audience)`** — fetches the tenant's JWKS (cached in memory), finds the key matching the token's `kid`, and validates the RS256 signature, audience, and issuer. Raises an exception on any failure.

**Usage:**

```python
from py_auth import verify_jwt

payload = verify_jwt(token, domain=AUTH0_DOMAIN, audience=AUTH0_AUDIENCE)
```

## Authentication

All protected routes use one of two decorators defined in `auth_routes.py`.

### `@require_auth`

Validates the Bearer token in the `Authorization` header and attaches the decoded JWT payload to `request.user_payload`. Raises `UnauthorizedError` on any failure (missing header, invalid token, expired token).

### `@roles_required(*roles)`

Wraps `@require_auth`. After JWT validation, it looks up the employee record, checks approval status, and verifies their role is in the allowed list. Raises:

- `EmployeeNotFoundError` — if no employee record or role exists for this Auth0 ID.
- `ForbiddenError` — if the account is not yet approved, or the role is not in the required list.

**Usage:**

```python
@auth_bp.route('/users/pending', methods=['GET'])
@roles_required('super-admin', 'hr')
def get_pending_users():
    ...
```

## Docker

The Docker build context is the **monorepo root** (`../..` relative to `apps/intranet/`), so the shared libraries in `libs/` are available during the build.

The Dockerfile copies shared libraries into the container before running `pip install`, so the relative paths in `requirements.txt` resolve correctly:

```dockerfile
COPY libs/py_auth    /opt/py_auth
COPY libs/py_errors  /libs/py_errors
COPY libs/py_config  /libs/py_config
COPY apps/intranet/backend/requirements.txt ./requirements.txt
RUN pip install --no-cache-dir -r requirements.txt \
    && pip install --no-cache-dir /opt/py_auth
```

`requirements.txt` references `py_errors` and `py_config` as editable local installs:

```
-e ../../../libs/py_errors
-e ../../../libs/py_config
```

With `WORKDIR /app`, `../../../libs/py_errors` resolves to `/libs/py_errors` — exactly where the `COPY` placed them. `py_auth` is installed separately after `requirements.txt` because it lives in `/opt/py_auth`.

To rebuild and start the stack:

```bash
# from apps/intranet/
docker compose up -d --build
```

## Module: Authentication & User Management (Auth)

### 1. Overview & Business Logic
The Auth module is the gatekeeper of the intranet. It seamlessly integrates **Auth0 Universal Login** with our custom PostgreSQL Database to provide strict Role-Based Access Control (RBAC). 

**The User Onboarding Journey:**
1. **Login Gateway:** New users authenticate via Auth0.
2. **Onboarding:** First-time users are prompted to enter their First and Last Name, and must select their **Shift Type** (`standard` or `extended`) and **Shift Hours** (`7am-5pm`, `9am-7pm`, or `custom`).
   - A `standard` shift dictates a 10-hour working window (600 minutes).
   - An `extended` shift dictates a 9.5-hour working window (570 minutes).
   - If `custom` shift hours are selected, the user must provide a valid `custom_shift_start` and `custom_shift_end` matching the exact expected duration.
3. **Pending Room:** The user's account is created in our database but flagged as `is_approved=False`. They are placed in a "Waiting for Approval" screen.
4. **HR Approval:** An HR or Super Admin assigns the user a Role and Department via the HR Dashboard, switching their account to `is_approved=True`. The user now has full system access.

### 2. Database Models (SQLAlchemy used as our ORM)
This module owns the core identity tables:

* **`Role`:** Defines security access tiers (e.g., `super-admin`, `hr`, `staff`).
* **`Department`:** Defines organizational units (e.g., `Software Development`, `Sales`).
* **`Employee`:** The central user profile.
  * *Architectural Note on Nullability:* In the `Employee` model, `role_id` and `department_id` are intentionally set to `nullable=True`. This is a business requirement because an employee record is created *during* the initial onboarding phase, but they do not receive a role or department until an HR admin explicitly assigns them one later.
  * *Standard Shift:* Defines the employee's expected working hours (e.g., `morning`, `afternoon`). This drives the automated attendance check-in rules.
### 3. Backend API Endpoints (Flask)
All endpoints require a valid Auth0 JWT token passed in the `Authorization: Bearer <token>` header.

* `POST /api/v2/auth/register`: Creates a new unapproved `Employee` record.
* `POST /api/v2/auth/login`: Checks user status. Blocks unapproved users with a `403 Forbidden`.
* `GET /api/v2/auth/me`: Returns the authenticated user's profile, including their stringified role and department.
* `GET /api/v2/auth/metadata`: *(Admin Only)* Returns a dictionary of all available roles and departments for dropdown menus.
* `GET /api/v2/auth/users/pending`: *(Admin Only)* Returns a list of all users awaiting approval.
* `POST /api/v2/auth/users/<user_id>/approve`: *(Admin Only)* Accepts a payload with `role_id` and `department_id`, approving the user and granting them access.

### 4. Frontend Components (Angular)
Located in `apps/intranet/src/app/features/`:

* **`LoginComponent`**: The public-facing landing page that triggers the Auth0 redirect.
* **`OnboardingComponent`**: Captures a new user's name and calls `/auth/register`.
* **`PendingComponent`**: A waiting room screen that periodically checks if the user has been granted access yet.
* **`EmployeesComponent`**: The shared HR/Super Admin Dashboard. It handles both active employees and pending user registrations. It utilizes `forkJoin` to simultaneously fetch pending users and system metadata, allowing admins to rapidly assign roles and approve accounts in one click.

### 5. Security & Permissions
* **Identity Provider:** Passwords and credentials are never stored in our database; they are handled securely by Auth0.
* **Token Verification:** Every backend request passes through a `verify_jwt` utility that validates the Auth0 token signature before executing logic.
* **Route Guards:** Sensitive backend routes are protected by a custom `@roles_required('super_admin', 'hr')` decorator. If a standard employee attempts to access an admin route, the request is instantly rejected with a `403 Forbidden` error.

## Module: Attendance Management (Attendance)

### 1. Overview & Business Logic
Tracks work hours for employees. Provides a clock-in and clock-out gateway on the homepage, showing active shift duration and timesheet history. The system uses the employee's `standard_shift` profile setting to dynamically enforce check-in cutoff times (e.g., blocking check-ins past 1:00 PM for Morning workers, and 5:00 PM for Afternoon workers). Restricts manual entries or corrections to HR and Super Admins (which also generates audit trails).

### 2. Database Models (SQLAlchemy)
- **`AttendanceRecord`**: Stores individual work sessions. Tracks `clock_in`, `clock_out` (null if active), `work_date`, `status` (e.g. `clocked_in`, `present`), `source` (e.g. `web`, `manual`), and `notes`.
- **`AuditLog`**: Stores audit trails of corrections made by HR or Super Admins. Tracks the editor (`changed_by`), the updated record (`record_changed`), previous timestamps (`previous_clock_in`, `previous_clock_out`), the `reason_for_change`, and the change timestamp (`changed_at`).

### 3. Backend API Endpoints (Flask)
All endpoints require a valid Auth0 JWT token passed in the `Authorization: Bearer <token>` header.
- `POST /api/v2/attendance/clock-in`: Clock in. Starts a new active `AttendanceRecord`.
- `POST /api/v2/attendance/clock-out`: Clock out. Updates the active `AttendanceRecord` with `clock_out` time and status.
- `GET /api/v2/attendance/me`: Returns paginated attendance record history for the authenticated employee.
- `GET /api/v2/attendance/stats`: Returns weekly work stats (hours this week) for the authenticated employee.
- `GET /api/v2/attendance`: *(Admin Only)* Returns a filtered, paginated list of all employees' attendance records.
- `POST /api/v2/attendance`: *(Admin Only)* Manually creates an attendance record.
- `PUT /api/v2/attendance/<uuid:id>`: *(Admin Only)* Modifies an attendance record and automatically appends an `AuditLog` entry.
- `GET /api/v2/attendance/<uuid:id>/audit`: *(Admin Only)* Fetches the audit trail for a specific attendance record.

### 4. Frontend Components (Angular)
- **`HomeComponent`**: Hosts the main clock-in/out form interface, displays the elapsed shift timer for the active session, handles check-in/out note input, and shows the personal timesheet history. Dynamically reads the user's `standard_shift` to determine when to trigger the Check-In popup (e.g., 8:00 AM vs 12:00 PM).
- **`AttendanceService`**: Communicates with the attendance API for clock operations, stats, and history retrieval.

## Module: Department Management (Departments)

### 1. Overview & Business Logic
Manages the company's organizational structure and reporting lines. Features include:
- **Current Departments:** The company structure includes Sales, Marketing, Operations (with sub-departments: Contact Center, Business Efficiency, Cloud & Automation, Software Development, Artificial Intelligence), Finance, HR, and IT.
- **Nesting/Hierarchy:** Departments can be nested under parent departments to form a scalable org chart.
- **Automated Permissions:** Assigning or removing a department manager automatically grants or revokes their `is_manager` permission flag.
- **Safeguards:** Restricts deleting departments that still contain active employees, and prevents managers from losing their role if they still manage other departments.

### 2. Database Models (SQLAlchemy)
- **`Department`**: Defines department metadata (`name` - stored normalized e.g., `hr`, `software_development`), `manager_id` (referencing the `Employee` model), and `parent_department_id` (self-referential foreign key for nesting).

### 3. Backend API Endpoints (Flask)
All endpoints require a valid Auth0 JWT token.
- `GET /api/v2/departments`: Lists all departments and their hierarchy. (Available to all roles)
- `POST /api/v2/departments`: *(HR & Super Admin)* Creates a new department. Automatically grants the manager role if a manager is assigned.
- `PUT /api/v2/departments/<int:id>`: *(HR & Super Admin)* Updates department details, parent hierarchy, and automatically syncs manager roles.
- `DELETE /api/v2/departments/<int:id>`: *(HR & Super Admin)* Deletes a department. Blocks deletion if active employees are assigned.

### 4. Frontend Components (Angular)
- **`DepartmentsComponent`**: A standalone, company-wide page providing a read-only org chart to all staff. HR and Super Admins see elevated UI controls (Edit drawers) to manage the structure.

## Module: Employee Management (Employees)

### 1. Overview & Business Logic
Enables HR and administrators to view, search, filter, edit, and deactivate/approve employee profiles.

### 2. Database Models (SQLAlchemy)
- **`Employee`**: The profile model storing first name, last name, email, role/department, and boolean flags for `is_approved`, `is_active`, and `is_internal`.

### 3. Backend API Endpoints (Flask)
- `GET /api/v2/employees`: *(Admin Only)* Lists and searches employees with query filters for `search` terms (name/email), `department_id`/`department_name`, `is_active`, `is_approved`, and `role`.
- `POST /api/v2/employees`: *(Admin Only)* Directly creates an employee profile (e.g. for pre-registration/lazy linking).
- `GET /api/v2/employees/<uuid:id>`: *(Admin Only)* Retrieves a single employee's details.
- `PUT /api/v2/employees/<uuid:id>`: *(Admin Only)* Updates employee details.
- `POST /api/v2/employees/<uuid:id>/deactivate`: *(Admin Only)* Deactivates an employee profile (`is_active=False`).
- `POST /api/v2/employees/<uuid:id>/approve`: *(Admin Only)* Approves a pending employee profile.

### 4. Frontend Components (Angular)
- **`EmployeesComponent`**: Administrative interface for managing the employee roster and approving pending registrations.

## Module: Reports & Dashboard (Reports)

### 1. Overview & Business Logic
Generates analytics reports and high-level KPIs for HR and administrators.

### 2. Backend API Endpoints (Flask)
- `GET /api/v2/reports/dashboard`: *(Admin Only)* Returns overall organization KPIs (total headcount, presence count, clock-in count, absent count, and today's attendance rate).
- `GET /api/v2/reports/attendance`: *(Admin Only)* Returns detailed employee attendance statistics. Supports CSV export by passing `format=csv` in the query string.
- `GET /api/v2/reports/attendance/departments`: *(Admin Only)* Returns aggregated attendance metrics across all departments.
- `GET /api/v2/reports/attendance/departments/<int:id>`: *(Admin Only)* Returns aggregated attendance metrics for a specific department.
